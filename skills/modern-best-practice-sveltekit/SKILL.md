# Writing SvelteKit Applications
 
We're using modern SvelteKit (with Svelte 5+ runes), and we're following common
best practices focused on clarity, correctness, and maintainability.
 
This skill covers SvelteKit-specific concerns (routing, data loading, server
code, deployment). For component-level concerns inside `.svelte` files, see
the `modern-best-practice-svelte-components` skill.
 
## Routing & File Structure
 
- **PREFER** colocating route-specific code with the route: if a helper,
  type, or server util is only used by `/routes/posts/[id]/`, put it there,
  not in `$lib`
- **USE** `$lib` for genuinely shared code (components, utilities, types,
  server helpers). Prefer `$lib/server/` for anything server-only — SvelteKit
  will error at build time if it's imported into client code
- **PREFER** layout groups `(group)` to share layouts without affecting URLs,
  rather than deeply nested `+layout.svelte` files that couple unrelated
  sections
- Keep route params typed — use the `PageLoad`/`PageServerLoad` types
  generated by SvelteKit (`import type { PageServerLoad } from './$types'`)
  rather than hand-typing them
- **AVOID** putting business logic in `+page.svelte`. Pages should render;
  loaders should load; actions should mutate.
## Rendering Modes
 
SvelteKit renders every route on the server by default, then hydrates. You
can override this per-route. Choose deliberately:
 
- **SSR (default)** — best for dynamic, personalized, or SEO-sensitive pages.
  First paint is fast; content is indexable.
- **Prerender** (`export const prerender = true`) — best for pages whose
  content doesn't depend on the request (marketing pages, docs, blog posts).
  Output is static HTML; zero server cost.
- **CSR-only** (`export const ssr = false`) — best for heavily interactive
  pages where SSR adds no value (dashboards behind auth, complex editors).
  Reach for this only when SSR actively hurts; don't disable by default.
- **SPA mode** (`adapter-static` with a fallback) — the whole app as a static
  bundle. Use when you don't have a Node/edge runtime available, accepting
  the SEO and first-paint tradeoffs.
Tradeoff rules of thumb:
- If the page is public and content-driven → prerender
- If the page is dynamic and public → SSR
- If the page is dynamic and behind auth → SSR (still useful for fast first
  paint), or CSR-only if the page is highly interactive and SEO is irrelevant
- **AVOID** disabling SSR just because a third-party library "doesn't work on
  the server" — usually the fix is to guard the code with `if (browser)` or
  import the library inside `onMount`, not to disable SSR for the whole route
## Data Loading
 
SvelteKit has two kinds of load functions. Picking the right one is the
single most common source of confusion.
 
### Universal load (`+page.ts` / `+layout.ts`)
 
- Runs on the server during SSR, then **runs again in the browser** on
  client-side navigation
- Use for public data from external APIs, or anything safe to run in both
  environments
- **CANNOT** touch databases, filesystem, secrets, or session cookies
- Use SvelteKit's provided `fetch` (from the load event) — it deduplicates
  and forwards cookies during SSR
### Server load (`+page.server.ts` / `+layout.server.ts`)
 
- Runs **only on the server**, ever
- Required for database queries, secrets, private API keys, filesystem
  access, or anything reading from the session/cookies
- Return value is serialized (via devalue) and sent to the client — keep it
  JSON-ish; don't return class instances or functions
### Rules of thumb
 
- **If it touches secrets or a database, it's a server load.** No exceptions.
- **AVOID** writing a universal load that `fetch()`es your own `+server.ts`
  endpoint. That's a pure-overhead HTTP hop — skip the endpoint and call the
  database directly from a server load.
- **PREFER** parallel data loading — return unawaited promises from load and
  let the page `{#await}` them, rather than awaiting everything sequentially
- For slow-but-non-critical data, return the promise itself (unawaited) from
  a server load to **stream** it to the client — the page renders
  immediately with the fast data and the slow data fills in when ready.
  Reach for this only when you have a genuinely slow query blocking an
  otherwise fast page.
- **USE** `depends()` and `invalidate()` for fine-grained refetching instead
  of full-page reloads
- **PREFER** `+layout.server.ts` for data that multiple child routes need
  (e.g. current user). Child loads automatically receive parent data via
  `await parent()`.
## Mutations: Form Actions vs API Routes
 
For changing data, SvelteKit gives you two options. Defaulting to the wrong
one is the second most common mistake.
 
### Form actions (`+page.server.ts` → `export const actions`)
 
- The idiomatic SvelteKit way to handle mutations tied to a page
- Work without JavaScript (progressive enhancement)
- Pair with `use:enhance` on the client for SPA-like UX with zero boilerplate
- Return data that's merged into the page's `form` prop
- Use for: create/update/delete from forms on the page, login, signup,
  settings changes — anything a user initiates from your own UI
### API routes (`+server.ts`)
 
- Raw HTTP endpoints returning JSON (or anything else)
- Use for: webhooks, third-party integrations, endpoints consumed by
  non-SvelteKit clients (mobile apps, other services), file uploads with
  special handling, server-sent events
### Rules of thumb
 
- **DEFAULT to form actions** for any mutation triggered from your own UI
- **AVOID** the pattern of `+server.ts` + client-side `fetch('/api/...', { method: 'POST' })`
  for internal mutations. It's more code, loses progressive enhancement, and
  you have to hand-roll what `use:enhance` gives you for free.
- Reach for `+server.ts` only when you genuinely need a real HTTP endpoint
  for an external consumer
## Auth & Sessions
 
- **ALWAYS** validate sessions on the server. Client-side checks are UX, not
  security — an attacker can trivially bypass them.
- **USE** `hooks.server.ts` to resolve the session once per request and
  attach it to `event.locals` (typed in `app.d.ts`). Every server load and
  form action can then read `locals.user` without re-checking.
- **PREFER** HTTP-only, `Secure`, `SameSite=Lax` cookies for session tokens.
  Set them via `event.cookies.set()`, never expose them to client JS.
- **DO NOT** put user objects in page data unless the page needs them.
  Leaking more user fields than necessary expands your attack surface.
- **USE** redirect-based auth guards in layout server loads:
  ```ts
  // +layout.server.ts for protected routes
  export const load: LayoutServerLoad = async ({ locals }) => {
    if (!locals.user) throw redirect(303, '/login');
    return { user: locals.user };
  };
  ```
- **AVOID** hand-rolling auth from scratch unless you have a reason.
  Session management, CSRF, token rotation, and secure cookie handling are
  easy to get subtly wrong. Prefer a maintained library or hosted provider.
## Environment Variables
 
SvelteKit has four env modules. Picking the wrong one leaks secrets or
breaks the build.
 
- **`$env/static/private`** — server-only, inlined at build time. Use for
  secrets that don't change between deploys (API keys, DB URLs).
- **`$env/static/public`** — client-safe, inlined at build time. Must be
  prefixed with `PUBLIC_`. Use for things like analytics IDs.
- **`$env/dynamic/private`** — server-only, read at runtime. Use when the
  value can change without rebuilding (e.g. per-environment config injected
  by the host).
- **`$env/dynamic/public`** — client-safe, read at runtime. Rare — prefer
  static public when possible.
Rules:
- **PREFER static over dynamic** unless you specifically need runtime values.
  Static is faster and catches typos at build time.
- **NEVER** import from `$env/static/private` or `$env/dynamic/private` in a
  file that could be bundled for the client. SvelteKit will error, but
  double-check when refactoring.
- **DO NOT** use `process.env` directly — it bypasses SvelteKit's validation
  and won't work on edge runtimes.
## Hooks
 
- **USE** `hooks.server.ts` for cross-cutting server concerns: session
  resolution, logging, request-scoped context, security headers
- **PREFER** a single `handle` function that composes sub-handlers via
  `sequence()` over one sprawling handler
- **USE** `handleFetch` to rewrite internal `fetch` calls during SSR (e.g.
  point them at an internal URL instead of public). Don't hand-roll this.
- **USE** `handleError` to log server errors to your observability tool.
  Return a safe message — don't leak stack traces to clients in production.
- **AVOID** doing heavy work in hooks. They run on every request; keep them
  fast and delegate to route-specific code where possible.
## Adapters & Deployment
 
- Pick the adapter that matches your runtime target: `adapter-node` for a
  long-running Node server, `adapter-vercel`/`adapter-netlify`/`adapter-cloudflare`
  for those platforms, `adapter-static` for fully static sites
- **BE AWARE** of runtime constraints: edge runtimes (Cloudflare Workers,
  Vercel Edge) don't have full Node APIs. `fs`, native modules, and some
  libraries won't work. Check before choosing.
- **USE** `adapter-auto` only for quick prototypes. For production, pin the
  specific adapter so your deploy target is explicit.
- **DO NOT** rely on in-memory state between requests in serverless
  deployments — there's no guarantee the same instance handles the next
  request. Use a real store (Redis, DB, KV).
## Error Handling
 
- **USE** `+error.svelte` for route-level error UIs. Place one at the layout
  level to catch errors from any child route.
- **THROW** `error(status, message)` from loads and actions for expected
  errors (404, 403, 400). SvelteKit renders the nearest `+error.svelte`.
- **THROW** `redirect(status, location)` for redirects. 303 for after-POST,
  307/308 for preserving method.
- **DO NOT** return error objects from loads; throw them. Returning
  `{ error: '...' }` makes every caller handle it manually.
- For form actions, **return** validation errors via `fail(400, { ... })`
  (don't throw) so the form prop is populated and the user's input survives
## General Principles
 
- Let the framework do its job — SvelteKit's conventions exist to solve
  real problems; fighting them usually means reinventing something worse
- Push logic toward the server when there's a choice — it's faster, safer,
  and reduces client bundle size
- Optimize for the request lifecycle: load in parallel, stream where it
  helps, cache at the right layer
- If a pattern feels complex, reconsider whether you're using the right
  primitive (load vs action vs endpoint vs hook)
